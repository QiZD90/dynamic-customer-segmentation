# Dynamic Customer Segmentation

Микросервис для работы с сегментами пользователей

## Используемые технологии

- net/http (в качестве веб-сервера)
- chi (в качестве HTTP router)
- PostgreSQL (в качестве БД)
- golang-migrate/migrate (для миграций БД)
- zerolog (для логгирования)
- Docker (для запуска сервиса)

## Usage

Запустить сервис можно командой:

```bash
 $ docker compose up --build
```

Как только сервер будет запущен, вам будет доступна Swagger документация по адресу

`http://localhost:80/swagger/index.html`

 (если стоит порт по умолчанию)

## Примеры запросов

### Создание сегмента

```bash
curl --request POST --url 'http://localhost:80/api/v1/segment/create' \
--header "Content-Type: application/json" \
--data '{
    "slug": "AVITO_TEST_SEGMENT"
}'
```

Ответ:

```json
{
    "status": "OK"
}
```

### Удаление сегмента


## Принятые решения

В ходе разработки были приняты следующие решения по вопросам, не обговорённым в ТЗ.

### Как реализовать авторизацию?

Согласно подходу Zero Trust, доверие к запросам не может быть связано с
местоположением и построением сети, поэтому самого факта того, что
микросервис будет недоступен за пределами корпоративной сети недостаточно для
обеспечения безопасности. Любые запросы должны выполняться исключительно по
токену авторизации, выданному микросервисом, отвечающим за аутентификацию и
авторизацию пользователей. Однако, поскольку разработка подобного сервиса выходит
за scope задачи, было принято решение допустить упрощение и пренебречь авторизацией.

### Откуда брать список пользователей?

Во всём сервисе, какая-либо информация о пользователях нужна лишь в функции
добавления нового сегмента запрошенному проценту пользователей, которая требует
получения списка ID существующих пользователей. Это не оправдывает необходимость
в хранении и поддержании актуальной информации о пользователях в самом сервисе,
тем более, что в реальном случае эта информация уже хранится и используется в
других местах. Поэтому было решено реализовать получение этого списка следующим
образом:

- В самом сервисе создан независящий от реализации интерфейс, позволяющий
запрашивать ID определённого процента пользователей
- Создана реализация данного интерфейса, запрашивающая эту информацию у
микросервиса с базой данных пользователей через REST API
- Создана моковая реализация подобного микросервиса для демонстрации работы проекта
- Остаётся возможность с минимальными изменениями кодовой базы хранить информацию
о пользователях в БД самого сервиса и получать её через другую реализацию интерфейса

### Где хранить CSV файл отчёта?

Было принято решение создать независимый от реализации интерфейс по сохранению
файла, позволяющий использовать интеграцию с любым внешним или внутренним ресурсом по хранению
файлов. Приведён пример реализации, хранящий файлы на диске на сервере и отдающий
их по HTTP запросу
